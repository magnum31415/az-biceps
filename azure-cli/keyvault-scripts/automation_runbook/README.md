# Azure Key Vault Backup Automation - Architecture

## 1. Overview

This solution implements an automated backup mechanism for Azure Key Vault using Azure Automation.

The goal is to:
- Periodically back up Key Vault secrets, keys, and certificates
- Store backups securely in Azure Storage
- Use Managed Identity for secure, passwordless authentication
- Ensure idempotent and repeatable deployment via scripting

This diagram shows:

- **Deployment Plane**  
  `deploy-automation.sh` creates all resources

- **Execution Plane**  
  `Schedule → Runbook`

- **Identity Plane**  
  Managed Identity accesses Key Vault and Storage

- **Data Plane**  
  Backups → Blob Container

# 2. Architecture Components
## 2.1 Azure Automation Account
Hosts the PowerShell runbook
Executes the backup process on a schedule
Uses System Assigned Managed Identity for authentication
## 2.2 Managed Identity
Automatically created and assigned to the Automation Account
Used to authenticate against:
Azure Key Vault
Azure Storage Account
## 2.3 Azure Key Vault
Source of the backup
Contains:
Secrets (excluding certificate-backed secrets)
Keys
Certificates
## 2.4 Azure Storage Account
Destination of the backups
Stores .bak files generated by Key Vault backup operations
Organized using a structured path: <kvName>/<timestamp>/<artifact>.bak

# 4. Backup Strategy

Azure Storage provides a modern, policy-driven approach to data protection, where traditional backups are replaced by native platform capabilities that ensure durability, recoverability, and cost efficiency.

In modern Azure architectures, traditional "backup" approaches are no longer the primary mechanism for data protection. Instead, Azure provides built-in capabilities within Storage Accounts that collectively replace classical backup solutions.

Storage tiers such as Cool and Archive are used to optimize cost, making them suitable for backup and long-term retention scenarios.

## Modern Backup Approach

The concept of backup in Azure is implemented through a combination of the following features:

- Versioning
Maintains previous versions of objects, enabling recovery from accidental changes or deletions.
- Soft Delete
Retains deleted data for a defined period, allowing restoration without permanent data loss.
- Immutability
Protects data from modification or deletion for a specified retention period (WORM – Write Once Read Many).
- Replication
Ensures data durability and availability across zones or regions.
- Lifecycle Management
Automatically moves or deletes data based on defined policies to optimize cost.

**Recommended Configuration**
-✔️ Versioning → Enabled
-✔️ Soft Delete → 30–90 days
-✔️ Immutability → Minimum 30 days
-✔️ Lifecycle Policies → Enabled
-✔️ Replication → ZRS or GRS

**Scope of Configuration**
````
Storage Account
├── Global configuration (versioning, soft delete, replication, etc.)
└── Containers (data only)
````
# 5. Commands Summary
**Key Vault**
````bash
# Verify if RBAC is enabled
az keyvault show \
  --name kv-test-ricard-001 \
  --query properties.enableRbacAuthorization
````
**Storage Configuration**
````bash
az storage account blob-service-properties update \
  --account-name $STORAGE_ACCOUNT \
  --resource-group $RESOURCE_GROUP \
  --enable-versioning true \
  --enable-delete-retention true \
  --delete-retention-days 7
````
